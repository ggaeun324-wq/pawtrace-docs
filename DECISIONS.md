# 📐 DECISIONS — 기술 의사결정 기록 (ADR)

> **목적**: 주요 설계 결정의 **맥락·대안·선택 이유·트레이드오프**를 기록합니다.
> ADR(Architecture Decision Record) 형식을 따르며, 시니어/CTO가 "왜 그렇게 했는가"를
> 빠르게 검토할 수 있도록 작성했습니다.

---

## ADR 형식
각 결정은 **맥락(Context) → 결정(Decision) → 대안(Alternatives) → 트레이드오프** 로 정리합니다.

---

## ADR-001 — 컨테이너 오케스트레이션: ECS Fargate (vs EKS)

- **맥락**: 1인 개발, 운영 부담 최소화, 그러나 프로덕션급 패턴 학습 필요.
- **결정**: **ECS Fargate** 채택 (서버리스 컨테이너).
- **대안**: EKS(Kubernetes) — 기능은 풍부하나 컨트롤플레인 비용·운영 복잡도가 큼.
- **트레이드오프**: K8s 생태계 경험은 줄지만, **노드 관리 없이** 핵심(네트워킹·IAM·배포)에 집중. 향후 EKS 확장 여지를 아키텍처에 남겨둠.

## ADR-002 — 인증: OIDC 키리스 (vs 장기 액세스키)

- **맥락**: GitHub Actions가 AWS에 접근해야 함.
- **결정**: **OIDC 페더레이션**으로 단기 자격증명 사용.
- **대안**: IAM 액세스키를 GitHub Secrets에 저장 — 가장 흔하지만 **장기 비밀이 노출되면 위험**.
- **트레이드오프**: 초기 신뢰정책 설정이 더 복잡하지만, **저장된 비밀이 없어** 보안이 근본적으로 우수. 현업 표준과 일치.

## ADR-003 — IAM 역할 분리 (단일 역할 ❌)

- **맥락**: CI가 앱 배포와 인프라 변경을 모두 수행.
- **결정**: **배포 역할 / 인프라 역할**을 분리하고 각각 최소 권한 부여.
- **대안**: 모든 권한을 가진 단일 역할 — 단순하지만 침해 시 **폭발 반경**이 큼.
- **트레이드오프**: 역할·신뢰정책 관리 포인트가 늘지만, **권한 경계가 명확**해 감사·통제가 쉬움.

## ADR-004 — 시크릿 관리: Secrets Manager (vs 환경변수 평문)

- **맥락**: DB 접속정보 등 민감값을 안전하게 주입해야 함.
- **결정**: **Secrets Manager**에 저장 → ECS가 런타임 주입, 실행 역할에 해당 시크릿만 허용.
- **대안**: Terraform 변수/이미지에 평문 — 간단하지만 **유출 위험**과 회전 어려움.
- **트레이드오프**: 약간의 비용/구성 추가. 대신 **코드·이미지에 비밀이 없음** + 회전 가능.

## ADR-005 — Terraform Remote State (S3 + DynamoDB Lock)

- **맥락**: 로컬 state 폴더 혼선으로 리소스 중복 생성 위험을 실제로 경험 ([TROUBLESHOOTING #4](./TROUBLESHOOTING.md)).
- **결정**: state를 **S3**에 저장, **DynamoDB**로 동시성 잠금.
- **대안**: 로컬 state — 1인에는 편하나 위치 의존·협업/CI 불가·사고 위험.
- **트레이드오프**: 부트스트랩 1회 구성 필요. 대신 **위치 독립·잠금·CI 연동** 확보.

## ADR-006 — 인프라 변경: GitOps (plan-on-PR / apply-on-merge)

- **맥락**: 콘솔 수작업은 추적·리뷰가 안 되고 재현 불가.
- **결정**: 모든 인프라 변경을 **PR → plan 코멘트 → 머지 시 apply**.
- **대안**: 콘솔 클릭 / 로컬 apply — 빠르지만 **이력·승인·재현성**이 없음.
- **트레이드오프**: 변경 속도는 약간 느려지지만, **모든 변경이 리뷰·기록**되고 Git이 진실원천.

## ADR-007 — 단일 NAT Gateway (vs AZ별 NAT)

- **맥락**: 포트폴리오 환경, 비용 민감.
- **결정**: **단일 NAT**로 비용 절감.
- **대안**: AZ별 NAT — 고가용성이 높지만 비용 약 2배.
- **트레이드오프**: NAT가 위치한 AZ 장애 시 아웃바운드 영향. **의도적 비용/가용성 트레이드오프**임을 문서로 명시(운영 전환 시 AZ별 NAT 권장).

## ADR-008 — 데이터 소스: 시드 데이터로 시작 (vs 처음부터 DB 연결)

- **맥락**: MVP를 빠르게 동작시키고, 인프라/배포 학습에 집중.
- **결정**: 초기에는 **시드 데이터**로 API가 즉시 동작하도록 함. Repository 레이어로 추상화해 **나중에 RDS로 무중단 교체** 가능하게 설계.
- **대안**: 처음부터 RDS 의존 — 로컬/CI 실행이 무거워짐.
- **트레이드오프**: 실DB 기능(쿼리·트랜잭션) 검증은 뒤로 미뤄짐. 대신 **개발 속도와 테스트 용이성** 확보, 교체 비용은 추상화로 최소화.

## ADR-009 — 보안의 파이프라인 내장 (Shift-Left)

- **맥락**: 사람이 매번 보안 점검을 기억할 수 없음.
- **결정**: **Trivy·SBOM·GitGuardian·SonarQube·Hadolint**를 CI/CD에 게이트로 내장.
- **대안**: 배포 후 점검/수동 검토 — 누락 가능, 늦은 발견.
- **트레이드오프**: 파이프라인이 다소 느려지고 가끔 false positive. 대신 **취약점/시크릿이 운영에 도달하기 전 차단**.

## ADR-010 — 표현의 중립성 (제품·법적 결정)

- **맥락**: 입양 투명성 서비스로서 특정 업체 단정은 법적/윤리적 리스크.
- **결정**: "불법/펫샵 확정" 같은 단정 대신 **"검증 필요 / 투명성 낮음 / 공공데이터 불일치"** 등 **중립적 지표**로만 표현.
- **대안**: 직접적 고발형 표현 — 주목도는 높으나 명예훼손 리스크.
- **트레이드오프**: 메시지가 부드러워지지만, **법적 안정성과 신뢰성**을 확보. 데이터 기반 플랫폼 포지셔닝과 일치.

## ADR-011 — 장바구니 영속화: PostgreSQL (vs Redis 세션)

- **맥락**: 장바구니는 '결제 전 담아둔 상태'라 기기/세션이 바뀌어도 유지돼야 함. 결제는 재고·쿠폰·후원과 함께 **정합성**이 중요.
- **결정**: 장바구니·주문·재고를 **같은 PostgreSQL**에 두어 하나의 트랜잭션으로 처리. 결제(checkout)는 **행 잠금(FOR UPDATE)** 으로 동시성 경합에서 **초과판매 0건**을 보장.
- **대안**: Redis 세션 장바구니(TTL) — 초고트래픽엔 유리하나 주문/재고와 **정합성 경계가 갈라짐**.
- **트레이드오프**: 초고트래픽 확장 카드(Redis 장바구니)는 뒤로 미루고, MVP는 **단순·정확**을 우선. 쓰기·동시성은 부하테스트의 핵심 워크로드로 검증.

## ADR-012 — 네트워크 보안 강화: WAF · HTTPS · VPC 엔드포인트

- **맥락**: '신뢰'를 파는 서비스가 인터넷에 노출된 ALB를 무방비로 두면 모순. 회원/결제 도입으로 공격면도 확대.
- **결정**: ALB 앞단 **WAF v2**(관리형 규칙 + IP rate-limit), **HTTPS/TLS1.3**(ACM 조건부 + HTTP→HTTPS 리다이렉트), AWS 서비스 접근은 **VPC 엔드포인트**로 프라이빗화. 앱 시크릿은 Secrets Manager JSON으로 주입.
- **대안**: 앱 레벨에서 직접 방어/HTTP 유지 — 운영 부담↑, 방어면↓, NAT로 모든 egress 통과.
- **트레이드오프**: 소폭 비용(WAF 월 고정비) 증가. 대신 **넓은 방어면·전송 암호화·경계 축소**를 적은 운영 부담으로 확보.
