# 🛠️ TROUBLESHOOTING — 실제 트러블슈팅 사례

> **목적**: 프로젝트를 진행하며 실제로 부딪힌 문제와 **원인 분석·해결 과정**을 기록합니다.
> 면접에서 "어떻게 디버깅하는 사람인가"를 보여주는 핵심 자료입니다.
> 각 사례는 *증상 → 원인 → 해결 → 배운 점* 구조로 정리했습니다.

---

## 사례 1 — ECR 푸시 시 "no basic auth credentials"

**증상**
Docker 이미지를 ECR에 push하려는데 인증 오류로 거부됨.

**원인**
ECR은 일반 Docker 레지스트리처럼 정적 비밀번호를 쓰지 않고, **AWS 자격증명으로 임시 토큰을 발급**받아 로그인해야 함. 그 로그인 단계를 거치지 않아 Docker가 자격을 모름.

**해결**
`aws ecr get-login-password`로 임시 토큰을 받아 `docker login`에 파이프 → 인증 후 push 성공. (CI에서는 `amazon-ecr-login` 액션이 이 과정을 대신함)

**배운 점**
ECR 인증은 **시간 제한 토큰** 기반. "레지스트리 로그인 = AWS 인증의 연장"이라는 점을 이해하니 CI에서 OIDC→ECR 흐름이 명확해짐.

## 사례 2 — `aws` 명령어를 "인식할 수 없음" (PATH 문제)

**증상**
AWS CLI를 설치했는데도 VS Code 터미널에서 `aws`가 인식되지 않음. 외부 PowerShell에서는 정상.

**원인**
**환경변수 PATH는 프로세스가 시작될 때 한 번 읽힘.** VS Code는 CLI 설치 *이전*에 켜져 있었기 때문에, 새 터미널을 열어도 부모 프로세스(VS Code)의 오래된 PATH를 물려받음.

**해결**
VS Code를 **완전히 종료 후 재시작** → 새 PATH를 읽어 정상 인식.

**배운 점**
"새 터미널 = 새 환경"이 아니다. 환경변수는 **프로세스 트리를 따라 상속**된다. 이 원리는 컨테이너 환경변수 주입을 이해하는 데도 그대로 적용됨.

## 사례 3 — Terraform "Inconsistent dependency lock file"

**증상**
`terraform apply` 실행 시 lock 파일 불일치 오류.

**원인**
프로바이더 의존성을 기록한 `.terraform.lock.hcl`과 실제 초기화 상태가 어긋남.

**해결**
`terraform init`(필요 시 `-upgrade`)으로 프로바이더를 재동기화 → 해결.

**배운 점**
lock 파일은 **재현 가능한 빌드**를 위한 장치. npm의 `package-lock.json`과 같은 역할임을 이해.

---

## 사례 4 — Terraform state 폴더 혼선 (가장 교훈적)

**증상**
부트스트랩 `apply` 시 "테이블이 이미 존재한다"며 **이미 만든 리소스를 또 만들려고** 함.

**원인**
실제 state 파일은 **중첩된 폴더**에 있었는데, 한 단계 바깥 폴더에서 apply를 실행함. 바깥 폴더의 state는 거의 비어 있어, Terraform이 "아무것도 없다 → 새로 만들자"라고 판단.

```
projects/pawtrace/pawtrace/.../bootstrap/terraform.tfstate   ← 진짜 (리소스 8개 기록)
projects/pawtrace/.../bootstrap/terraform.tfstate            ← 빈 state (사고 발생 지점)
```

**해결**
진짜 state 파일을 올바른 작업 폴더로 복사한 뒤 apply → 정상(중복 생성 없이 변경분만 반영).
이후 **remote state(S3 + DynamoDB Lock)** 로 전환해 폴더 위치에 의존하지 않도록 구조 개선.

**배운 점**
> **"Terraform에서 진실의 원천은 코드가 아니라 state다."**
state가 실제 리소스를 추적하므로, state를 잃거나 헷갈리면 인프라가 어긋난다.
그래서 실무는 **remote state**로 위치를 고정하고 잠금을 건다.

## 사례 5 — 프론트엔드 ↔ API 호출이 CORS로 차단

**증상**
브라우저에서 프론트엔드가 API를 호출하면 응답이 막힘 (CORS 정책 위반).

**원인**
API 서버가 **허용 오리진(origin) 목록**에 프론트엔드 주소를 갖고 있지 않음. 브라우저는 다른 오리진의 응답을 보안상 차단.

**해결**
허용 오리진을 **환경변수로 분리**하여 주입(코드 하드코딩 X) → 헤더(`Access-Control-Allow-Origin`)가 정상 반환되며 연결 성공.

**배운 점**
CORS는 **서버가 허용을 명시**해야 하는 브라우저 보안 모델. 오리진은 환경마다 다르므로 **설정으로 외부화**하는 것이 12-Factor 원칙과 일치.

## 사례 6 — IPv4/IPv6 (`localhost` vs `127.0.0.1`)

**증상**
헤드리스 브라우저 테스트에서 `localhost`로는 API 연결 실패, `127.0.0.1`로는 성공.

**원인**
`localhost`가 IPv6(`::1`)로 해석되는데 API 서버는 IPv4(`127.0.0.1`)에 바인딩되어 있어 불일치.

**해결**
테스트에서 명시적으로 `127.0.0.1`(IPv4)을 지정 → 연결 성공.

**배운 점**
`localhost`는 이름일 뿐, **IPv4/IPv6 중 무엇으로 해석되는지**가 환경마다 다르다. 네트워크 디버깅 시 이름이 아니라 **실제 바인딩 주소**를 확인해야 함.

## 사례 7 — VS Code에서 Git 푸시가 안 됨 (외부 PowerShell은 됨)

**증상**
PowerShell에서는 `git push`가 되는데 VS Code에서는 인증/푸시가 실패.

**원인**
VS Code가 사용하는 Git **자격증명 도우미/PATH 컨텍스트**가 외부 셸과 달랐음(역시 프로세스 환경 상속 문제 계열).

**해결**
환경(자격증명 도우미·PATH)을 일치시키고 VS Code를 재시작 → 정상 푸시.

**배운 점**
같은 명령도 **실행 컨텍스트(환경변수·자격증명 저장소)** 가 다르면 결과가 다르다. "도구가 아니라 환경을 의심하라."

## 공통 교훈 (Meta-Lessons)

| 패턴 | 교훈 |
|---|---|
| PATH·자격증명 문제 반복 | **환경은 프로세스 시작 시 결정**되고 트리를 따라 상속된다 |
| state·lock 문제 | **재현성**을 위한 장치(state, lock)를 신뢰하고 보호하라 |
| CORS·IPv6 | 증상이 아니라 **프로토콜/보안 모델의 원리**로 돌아가 진단하라 |
| 전반 | 임시방편보다 **구조 개선**(remote state, 환경변수화)으로 재발 방지 |
