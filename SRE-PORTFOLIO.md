## 1. 부하테스트 — 왜, 어떤 상황을, 얼마나

### [문제 · 의도]

PawTrace는 평소 트래픽은 잔잔하지만, **특정 순간에 트래픽이 한 페이지로 쏠리는** 서비스입니다.
그래서 "서버 전체를 골고루 때리는" 테스트가 아니라, **실제로 사람이 몰리는 순간을 그대로 재현하는** 시나리오를 설계했습니다. 세 가지 상황을 정의했습니다.

- **① 버스·옥외광고 집행일** — 광고를 본 사람들이 동시에 앱을 열고, 홈의 **'입양이 시급한 공고'(`/dogs/urgent`)** 첫 화면으로 몰립니다.
- **② 입양의 날(캠페인)** — 마감 임박 공고가 첫 페이지에 노출되며 **같은 첫 화면**에 트래픽이 집중됩니다.
- **③ 쇼핑몰 할인 오픈** — 정각에 **상품 목록(`/shop/products`)** 으로 '오픈런' 트래픽이 쏠립니다.

### [시나리오]

상황별로 **부하의 모양(프로파일)** 을 다르게 설계했습니다. (도구: k6, 대상: 실제 ALB 엔드포인트)

| 시나리오 | 재현 대상 | 부하 프로파일 | 확인 지표(SLO) |
|---|---|---|---|
| 평상시(기준선) | 전체 여정 | 동시 50명, 10분 유지 | p95 지연·에러율 기준값 확보 |
| ①② 첫페이지 스파이크 | `/dogs/urgent`·`/dogs/today` | **30초 만에 50→800명**, 3분 유지 후 급감 | 첫화면 p95 < 300ms, 에러율 < 1% |
| ③ 오픈런 스파이크 | `/shop/products` | 정각 0초에 동시 600명 순간 유입 | p95 < 400ms, 타임아웃 0건 |
| 한계 탐색 | 무거운 경로(지도 검색 `/shelters`) | 한계까지 계단식 증가 | **한계점·오토스케일 임계값 실측** |

### [해결 · 수치화]

한계 테스트에서 **지도 검색(`/shelters`)이 먼저 무너지는** 것을 확인했고, 원인은 보호소 하나마다 '입양 가능 수'를 세는 쿼리가 따로 나가는 **N+1 구조**였습니다.

1. **집계 쿼리 1회로 통합** — 보호소 목록과 강아지 수를 한 번에 계산
2. **Redis 캐싱** — 같은 지역을 반복 조회하는 트래픽을 DB 앞에서 흡수
3. **ECS 오토스케일** — 첫페이지 스파이크를 CPU 60% + ALB 요청수 기준으로 태스크 2→6 자동 확장

개선 후 **같은 시나리오를 재실행해 지도 검색 p95를 기준선까지 회복**시키는 것이 목표 수치입니다.
즉, 부하테스트는 "터뜨리기"가 아니라 **병목을 실측 → 처방 → 재검증으로 증명**하는 루프입니다.

## 2. 인프라 — 백엔드 구조에서 도출한 3-Tier 설계

### [설계]

FastAPI 백엔드가 `api → service → repository → DB`로 계층이 분리돼 있어, 이 논리를 **AWS 3-Tier로 그대로 매핑**했습니다.

```
인터넷
  │
  ▼  ── ① 웹(표현) : 퍼블릭 서브넷 ───────────────
[ CloudFront + S3 (Next.js) ]   [ ALB ]  ← 입구는 ALB 하나로 단일화
                                   │  /api/v1/health 상태 확인
  ▼  ── ② 앱(로직) : 프라이빗 서브넷 ─────────────
                          [ ECS Fargate (FastAPI) ]  ← ALB 거친 요청만 처리
  ▼  ── ③ 데이터 : 프라이빗 서브넷 ───────────────
     [ RDS PostgreSQL+PostGIS ]  [ ElastiCache Redis ]  [ S3 이미지 ]
```

| 계층 | AWS 리소스 | 배치 위치 | 이 자리에 둔 이유 |
|---|---|---|---|
| **① 웹(표현)** | CloudFront + S3(Next.js), **ALB** | 퍼블릭 서브넷 | 사용자 접점만 인터넷에 노출. **입구는 ALB 하나로 단일화** |
| **② 앱(로직)** | **ECS Fargate**(FastAPI) | 프라이빗 서브넷 | 외부에서 직접 못 닿게 숨김. **ALB를 거친 요청만** 처리 |
| **③ 데이터** | **RDS(PostgreSQL+PostGIS)**, **ElastiCache(Redis)**, S3(이미지) | 프라이빗 서브넷 | 가장 민감한 계층. **앱에서만** 접근 |

### [각 데이터 저장소를 이 자리에 둔 논리]

- **RDS (PostgreSQL + PostGIS)** — 보호소·강아지·이력은 **관계가 있는 핵심 데이터**라 관계형 DB를 선택했습니다. 지도 반경검색을 위해 공간 확장(PostGIS)을 쓰므로 **프라이빗 서브넷에 격리 + 암호화 + 외부 접근 차단**(`publicly_accessible=false`). 인터넷에서 DB로 가는 길 자체를 없앴습니다.
- **ElastiCache (Redis)** — 지도 검색처럼 **같은 결과를 반복 조회**하는 트래픽을 DB 앞에서 받아내는 캐시입니다. DB 부하를 덜어 첫페이지 스파이크를 버티는 핵심이라 앱 옆(프라이빗)에 배치했습니다.
- **S3 (이미지)** — 용량이 크고 정적인 이미지는 DB가 아니라 **객체 스토리지**로 분리(퍼블릭 접근 전면 차단)했습니다. DB는 가볍게 유지합니다.

### [계층 격리 방식]

계층 간 통신은 **보안그룹을 앞단끼리만 연결**(ALB→ECS→RDS/Redis)해, IP가 아니라 **'바로 앞 계층에서 온 요청만' 허용**하도록 구조로 강제했습니다. 가용성을 위해 **2개 AZ**로 이중화했습니다.

## 3. 관측성 — 느림을 '추측'이 아니라 '지목'으로

### [문제]

지도 검색 한 번은 `ALB → 앱(ECS) → DB(RDS)` 여러 구간을 거칩니다.
관측 장치가 없으면 "느리다"는 말은 들어도 **어느 구간이 범인인지** 알 수 없어 처방이 감(感)이 됩니다.

### [해결]

세 가지를 나눠서 보도록 했습니다.

- **사용자 체감** — 요청수·에러율·응답시간(p95)을 화면(홈·지도·이력)별로 수집해 SLO와 바로 비교합니다.
- **자원 상태** — ECS의 CPU/메모리, RDS 커넥션 수, Redis 적중률을 함께 봐서 **어디가 포화됐는지** 확인합니다(ECS Container Insights 활성화).
- **구간 추적** — 분산 추적(X-Ray)으로 요청을 구간별로 쪼개, "지도 검색 지연의 대부분이 **DB 집계 구간**"임을 눈으로 확인하고 앞의 N+1 처방으로 연결했습니다.

알람도 단순 임계치가 아니라 **에러 예산을 빠르게 소진할 때만** 울리게 해, 불필요한 새벽 호출(알람 피로)을 줄였습니다.

## 4. CI/CD · 보안

- **[GitOps 배포]** 앱 배포(`deploy.yml`)와 인프라 배포(`infra.yml`)를 **분리**하고, PR에 `terraform plan` 결과를 코멘트로 남긴 뒤 머지 시 apply하는 흐름으로 **인프라 변경도 코드 리뷰 대상**으로 만들었습니다.
- **[키리스 인증]** AWS 장기 액세스키는 유출 위험이 커서 **OIDC 키리스 인증**으로 전환(키 미저장)하고, 배포용·인프라용 IAM 역할을 분리(최소 권한)했습니다.
- **[시크릿·이미지 보안]** DB 비밀번호·API 키는 **Secrets Manager**에서 런타임 주입하고, 빌드 단계에서 **Trivy로 이미지 취약점**을 검사해 위험한 이미지는 배포를 막습니다. 업로드 이미지의 위치정보(EXIF)는 저장 전 제거합니다.

## 5. 설계 의도 · 트레이드오프

- 같은 컨테이너 이미지를 **ECS → EKS로 그대로 옮길 수 있게** 설계해 MVP의 단순함과 향후 확장성을 동시에 잡았습니다.
- 비용을 위해 NAT는 1개로 두되, egress 병목은 **S3 VPC 엔드포인트로 우회**하는 개선안을 부하테스트 결과와 함께 준비했습니다.
- 오토스케일 기준도 감이 아니라 **한계 테스트로 실측**해 CPU·요청수 기준으로 정했습니다.
